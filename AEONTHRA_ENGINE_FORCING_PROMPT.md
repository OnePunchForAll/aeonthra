## 1. THE SEMANTIC→DETERMINISTIC TRANSLATION PRINCIPLE

A language model succeeds at content generation because it performs three operations implicitly:

1. **Salience detection** — knowing what matters in a text
2. **Relational understanding** — knowing how ideas connect
3. **Plausibility judgment** — knowing when generated output makes sense

Your deterministic engine must perform all three explicitly using the techniques below. Every stage of your pipeline must tie back to one of these three operations.

### 1.1 Deterministic Salience Detection

An LLM "just knows" that "Utilitarianism" is more important than "Furthermore" in an ethics chapter. You don't have that. To approximate salience, you combine:

- **TF-IDF with logarithmic IDF** — measures term importance within a document vs across documents
- **Position weighting** — terms in titles, first paragraphs of sections, and near definition patterns get higher scores
- **Syntactic role scoring** — terms appearing as subjects of definition verbs ("is," "refers to," "means") score higher than terms appearing as objects of prepositions
- **Frequency distribution shape** — real concepts appear in multiple sections; noise appears in one
- **Co-occurrence density** — real concepts co-occur with other real concepts in the same paragraphs; noise has random co-occurrence
- **Structural markers** — terms that appear in bold, italic, headings, or definition lists get confidence boost
- **Morphological signals** — words ending in -ism, -ology, -ics, -ity, -tion, -ment are more likely to be concepts than verbs or function words

Every candidate concept must score across ALL of these dimensions. A single score from a single method is insufficient. Final salience score is a weighted ensemble.

### 1.2 Deterministic Relational Understanding

An LLM "just knows" that "Bentham's calculus" and "Mill's higher pleasures" are related. You must derive relationships from observable textual patterns:

- **Direct mention** — Concept A's definition contains Concept B's name
- **Paragraph co-occurrence** — Both appear in the same paragraph N or more times
- **Contrastive markers** — "unlike X, Y…", "in contrast to", "whereas", "X vs Y", "differs from"
- **Causal markers** — "because", "therefore", "as a result", "leads to", "follows from"
- **Hierarchical markers** — "a type of", "a form of", "is a kind of", "falls under", "extends"
- **Temporal markers** — "developed from", "replaced by", "built upon", "evolved into"
- **Keyword Jaccard** — overlap in non-stopword vocabulary between concept definitions
- **Shared philosopher/author** — concepts attributed to the same source are likely related

Each relationship type uses a specific detection algorithm. You will implement all eight.

### 1.3 Deterministic Plausibility Judgment

An LLM rejects nonsense output because nonsense doesn't match its learned distributions. You must validate output against explicit quality gates:

- **Grammatical structure check** — does the generated sentence have a subject and a verb?
- **Length distribution check** — is the output within the expected character/word range for its type?
- **Lexical diversity check** — are the words varied enough (unique/total ratio above threshold)?
- **Stopword density check** — is the output at least 40% content words (not just function words)?
- **Coherence check** — does the output share vocabulary with its intended source?
- **Plausibility check** — for wrong answers, is the wrong answer a real claim about something (even if wrong about THIS concept), not gibberish?
- **Diversity check** — is the output sufficiently different from already-accepted output in the same batch (minimum edit distance)?

Any output that fails a single gate is rejected and regenerated. There is no "maybe good enough." There is passed or rejected.

## 2. FORBIDDEN SHORTCUTS

You will not do any of the following. Each is a shortcut that produces the garbage output I'm trying to eliminate.

❌ **Do not** use a generic "list of stopwords" as your only noise filter. You need content-aware block rejection.

❌ **Do not** bundle fewer than 10,000 English words in your frequency dictionary. A small dictionary produces a small vocabulary ceiling and destroys question quality.

❌ **Do not** bundle fewer than 3,000 synonym pairs. Synonyms are the backbone of paraphrase generation — too few means you'll regenerate the same sentence fragments.

❌ **Do not** use fewer than 20 mnemonic templates. Three templates produce identical-sounding mnemonics for every concept and the student will lose trust in the system within 5 concepts.

❌ **Do not** write fewer than 12 question-generation methods. Each method targets a different cognitive level and without variety the quiz becomes monotonous.

❌ **Do not** generate wrong answers by random word replacement. Wrong answers must be meaningfully wrong — they must describe a different REAL concept from the same chapter, not nonsense.

❌ **Do not** skip validation. Every generated artifact (concept, question, mnemonic, dilemma, corruption) must pass validation gates or be rejected and regenerated or dropped.

❌ **Do not** fabricate content when extraction yields too little. If a chapter produces fewer than 8 valid concepts, return an explicit "insufficient content" state. Never pad with junk.

❌ **Do not** use placeholder templates with raw variable names visible. No "[concept] is a type of [thing] that [does something]" outputs ever reach the UI.

❌ **Do not** generate content before the full source text has been normalized and segmented. Never run extraction on raw HTML.

❌ **Do not** use `Math.random()` to select template variants without tracking what's already been used. Random selection without deduplication produces collision within 4 generations.

❌ **Do not** treat LMS chrome (navigation, buttons, module labels, due dates, "mark done") as content. You must strip it before extraction.

## 4. THE 10-STAGE PIPELINE (ELABORATED)

This is the exact stage order. Each stage has explicit inputs, explicit outputs, and explicit quality gates. Do not merge stages. Do not skip stages. Do not reorder.

### STAGE 1: Text Ingestion and Chrome Stripping

**Input:** Raw HTML string OR plaintext from upload.

**Operations:**
1. If HTML, parse with DOMParser. Strip `<script>`, `<style>`, `<nav>`, `<header>`, `<footer>`, and elements matching chrome selectors (listed in Section 3.2 of AEONTHRA_CORRECTIVE.md).
2. Extract text with structure preservation: `<h1>`–`<h6>` → `[H1]...[/H1]`, `<li>` → `[LI] text`, `<strong>`/`<em>` tracked in a side map of "emphasized terms".
3. Decode HTML entities.
4. Normalize whitespace: collapse multiple spaces, preserve paragraph breaks as double newline.
5. Strip repeated artifacts: page numbers (`^\d+$` on own line), running headers that appear 3+ times, copyright notices.
6. Remove any text block where 60%+ of the tokens appear in `canvas-chrome.json`.

**Output:** `{ normalizedText: string, emphasizedTerms: Set<string>, structuralMap: StructuralMap }`

**Quality gate:** If `normalizedText.length < 500`, return `{ status: "insufficient-content", reason: "source too small after cleanup" }`. Stop pipeline.

### STAGE 2: Block Segmentation

**Input:** Stage 1 output.

**Operations:**
1. Split by `[H1]`/`[H2]` markers into top-level sections.
2. Within each section, split into blocks:
   - Each heading becomes a block with `type: "heading"`
   - Each paragraph (≥ 40 chars) becomes a block with `type: "paragraph"`
   - Each list item (≥ 20 chars) becomes a block with `type: "list-item"`
3. For each block, compute: `wordCount`, `sentenceCount`, `position`, `parentHeading`.
4. If no headings exist, infer section boundaries via topic shift detection:
   - Build term-frequency vector per paragraph
   - Compute cosine similarity between consecutive paragraphs
   - Where similarity drops below 0.25, insert a synthetic section boundary

**Output:** `Section[]` where each section contains `ContentBlock[]`.

**Quality gate:** If zero sections detected AND fewer than 5 paragraphs exist, return insufficient-content.

### STAGE 3: Block Quality Scoring

**Input:** Stage 2 blocks.

**Operations:** For each block, compute quality score using the exact formula:

```typescript
function scoreBlock(block: ContentBlock): number {
  let score = 0;
  const words = block.text.split(/\s+/);
  const wordCount = words.length;
  
  // Length scoring (sweet spot curve)
  if (wordCount < 12) return 0;
  if (wordCount > 400) score -= 10;
  if (wordCount >= 25 && wordCount <= 180) score += 15;
  else if (wordCount >= 15 && wordCount <= 250) score += 8;
  
  // Structural bonus
  if (block.type === "heading") score += 5;
  if (block.parentHeading) score += 3;
  
  // Sentence quality
  const sentences = block.text.split(/[.!?]+/).filter(s => s.trim().length > 0);
  if (sentences.length >= 2) score += 5;
  if (sentences.length >= 1) {
    const avgLen = wordCount / sentences.length;
    if (avgLen >= 8 && avgLen <= 25) score += 5; // natural sentence length
  }
  
  // Definition-like language (high-confidence learnable content)
  const defPatterns = [
    /\b(is|are)\s+(the|an?)\s+\w+/i,
    /\brefers\s+to\b/i,
    /\bmeans\s+that\b/i,
    /\bcan\s+be\s+defined\s+as\b/i,
    /\bis\s+defined\s+as\b/i,
    /\bthe\s+(view|idea|theory|belief|claim|principle|notion|concept)\s+that\b/i,
  ];
  const defMatches = defPatterns.filter(p => p.test(block.text)).length;
  score += defMatches * 12;
  
  // Example presence
  if (/\bfor\s+(example|instance)\b|\be\.g\./i.test(block.text)) score += 4;
  
  // Conceptual language density
  const conceptMarkers = /\b(theory|principle|concept|framework|approach|method|argument|claim|view|position|idea|notion|doctrine|thesis)\b/gi;
  const markerCount = (block.text.match(conceptMarkers) || []).length;
  score += Math.min(markerCount * 3, 15);
  
  // Chrome rejection (hard)
  const chromePatterns = [
    /^(module|item|quiz|assignment|due|points?|score|mark|done|complete)\b/i,
    /^(click|tap|select|choose|go to|return to|view|download|submit)\b/i,
    /\b(mark (as )?done|to do|complete this|check this)\b/i,
    /^[\d\s\W]*$/,
  ];
  if (chromePatterns.some(p => p.test(block.text))) return 0;
  
  // Pure navigation-label rejection
  if (wordCount <= 5 && /^[A-Z]/.test(block.text) && !/[.!?]$/.test(block.text)) return 0;
  
  return score;
}
```

**Output:** `Block[]` filtered to only blocks with `score >= 10`.

**Quality gate:** If fewer than 5 blocks pass, return insufficient-content.

### STAGE 4: Topic-Bearing Sentence Detection

**Input:** Stage 3 qualified blocks.

**Operations:** For each block, identify sentences that bear conceptual weight using all 40+ patterns from `definition-patterns.json`. Extract for each match:
- `candidateConcept` (the captured subject)
- `definition` (the captured predicate)
- `sourceBlock`
- `extractionMethod` and `confidence`

Additionally extract:
- **Contrastive sentences:** matching `/(unlike|in contrast to|whereas|while|as opposed to)\s+([^,]+),\s+([^.]+\.)/gi` — these become relationship data
- **Example sentences:** matching `/(for example|for instance|e\.g\.|such as),?\s+([^.]+\.)/gi` — these become evidence attachments
- **Characteristic sentences:** matching `/([A-Z][a-zA-Z\s]{2,40}?)\s+(involves|includes|requires|consists of|focuses on|emphasizes|is based on)\s+([^.]+\.)/g`

**Output:** `TopicSentence[]` with all extracted sentences, each tagged with their type and source.

**Quality gate:** If fewer than 8 topic-bearing sentences found, return insufficient-content.

### STAGE 5: Concept Candidate Generation (4-Method Ensemble)

**Input:** Stage 4 topic sentences + Stage 1 emphasized terms + Stage 2 headings.

**Run all four methods in parallel:**

**Method A: Explicit Definition Extraction (confidence 0.95–1.0)**
From topic sentences with definition patterns, each match produces a candidate. The captured concept name is normalized (trim whitespace, strip leading articles, fix case).

**Method B: Emphasized Term Extraction (confidence 0.85–0.95)**
From `emphasizedTerms` set. For each term, search all blocks for the sentence containing it. If that sentence has definitional structure, upgrade confidence to 0.95. Otherwise use the sentence as contextual definition.

**Method C: Heading-Derived Concepts (confidence 0.70–0.90)**
Each heading becomes a candidate IF the first 1-2 sentences under it form a definitional claim. Heading text → concept name. First sentences → definition.

**Method D: Statistical Key Phrase Detection (confidence 0.50–0.80)**

This is the most complex method. Implement it exactly as described:

1. Tokenize normalized text into words. Strip stopwords and chrome terms.
2. Extract n-grams of length 1, 2, 3 (unigrams, bigrams, trigrams).
3. Score each n-gram:
   - Frequency count across blocks (raw count)
   - Document frequency: number of distinct blocks it appears in
   - TF-IDF: `count × log(totalBlocks / documentFrequency)`
4. Apply morphological boost: n-grams ending in `-ism`, `-ology`, `-ics`, `-ity`, `-tion`, `-ment`, `-ance`, `-ence` get a ×1.5 multiplier.
5. Apply capitalization boost: n-grams that appear capitalized mid-sentence (proper nouns or titles) get ×1.3 multiplier.
6. Apply position boost: n-grams appearing in the first paragraph of a section get ×1.2 multiplier.
7. Filter: keep only n-grams with final score in top 8% OR raw frequency ≥ 5.
8. For each kept n-gram, find the first sentence in which it appears in a definitional context. If no such sentence exists, discard the candidate.

**Merge methods:**
After all four methods complete, merge results. For candidates that appear in multiple methods, combine confidences: `combined = 1 - (1-c1)(1-c2)(1-c3)(1-c4)` (probabilistic OR).

**Output:** `ConceptCandidate[]` with method-of-origin tracking and final confidence scores.

### STAGE 6: Canonicalization and Deduplication

**Input:** Stage 5 candidates.

**Operations:**

1. **Name normalization:**
   - Lowercase for comparison
   - Strip leading articles ("the", "a", "an")
   - Strip trailing punctuation
   - Stem variants: "utilitarian" and "utilitarianism" → canonical form is whichever scored higher

2. **Exact match dedup:** Merge candidates with identical normalized names, keeping the highest confidence and combining evidence.

3. **Subphrase absorption:** If candidate A is a substring of candidate B:
   - If both scored high AND B appears ≥ 3 times → keep B (more specific wins)
   - Else → keep A (more general wins)

4. **Jaccard-based fuzzy dedup:** For remaining candidates, compute character trigram Jaccard similarity pairwise. If similarity ≥ 0.85, merge.

5. **Rank by final score:**
```
finalScore = confidence × 100
           + evidenceCount × 10
           + distinctSourceBlocks × 8
           + (extractionMethod === "explicit-definition" ? 30 : 0)
           + (hasEmphasisOrigin ? 20 : 0)
           + (appearsInHeading ? 15 : 0)
           + morphologicalBoost × 10
```

6. Take top 12–18 candidates. If fewer than 10 remain after ranking, take what exists.

**Quality gate:** If fewer than 8 concepts survive, return insufficient-content.

### STAGE 7: Concept Enrichment

**Input:** Stage 6 ranked concepts.

For each concept, populate all fields:

**`core` (definition, cleaned):**
- Start with the best definition sentence (highest-confidence extraction)
- Strip leading filler: `/^(Basically|Essentially|In essence|Simply put|In short|To put it simply),?\s*/i`
- Trim to max 2 sentences using sentence splitter
- Ensure ends with period
- If > 35 words, truncate at nearest sentence boundary within the limit
- **Validation:** Must start with capital, end with period, contain at least one subject-verb pair (detected via simple parser: contains a noun followed by a verb from a 500-verb common verb list).

**`detail` (supporting context):**
- Sentence immediately following the core definition
- Plus any example sentence found in the same block (`"for example,..."`)
- Plus any elaboration sentence (starts with "This", "It", "The concept" and refers back)
- Max 3 sentences, max 80 words
- **Validation:** Must be distinct from `core` (Jaccard similarity ≤ 0.6 with core)

**`primer` (quiz context hint):**
- Generate via template: `"This relates to {subject_noun_from_core} — how it {verb_phrase_from_core}."`
- If template produces awkward output, fall back to first 15 words of `core`
- Max 15 words
- **Validation:** Must not end mid-word, must contain at least one content word (non-stopword)

**`mnemonic` (memory hook):**
- Select template from `mnemonic-templates.json` based on concept properties
- Selection priority:
  1. If concept has `opposites` → `contrast-anchor` template
  2. If concept name has 3+ words → `acronym-builder` template
  3. If concept has a philosopher association → `philosopher-dinner-party` template
  4. If definition contains a specific number → `number-hook` template
  5. If concept has a rhyme match in the rhyme dictionary → `rhyme-hook` template
  6. Else → cycle through remaining templates using concept ID as seed (not random) to ensure determinism
- Fill template slots from concept data
- **Validation:** Filled template must:
  - Contain no raw slot names (no `{...}`)
  - Be between 10 and 40 words
  - Not be identical to another concept's mnemonic in the same bundle (check via trigram Jaccard ≤ 0.7)
- If validation fails, try the next template in priority order
- If all templates fail, the concept's mnemonic is empty string (UI handles missing)

**`keywords`:**
- From `core` + `detail`, extract all content words (non-stopword, non-chrome)
- Keep only words appearing ≥ 2 times OR appearing in the concept name
- Limit to top 8 by frequency

**`relatedConcepts`:**
- Keyword Jaccard similarity vs all other concepts; include any with ≥ 0.25
- Plus any concept whose name appears in this concept's `detail`

**`opposites`:**
- Scan source text for contrastive structures near this concept
- Pattern: find sentences containing this concept + `unlike|contrast|whereas|opposed to|differs from|rather than`
- The other concept in the contrastive structure is an opposite

**`category`:**
- Use the parent heading of the concept's source block

**`confidence`:**
- Carry forward from Stage 5/6

### STAGE 8: Relationship Mapping

**Input:** Stage 7 enriched concepts + Stage 4 contrastive sentences.

For each ordered pair (A, B) of concepts, compute:

1. **Keyword Jaccard:** `|A.keywords ∩ B.keywords| / |A.keywords ∪ B.keywords|`. If ≥ 0.2 → add "overlaps-with" relationship.

2. **Paragraph co-occurrence:** Count paragraphs containing both names. If ≥ 2 → add "discussed-together" relationship.

3. **Explicit contrast:** If the pair appears in a Stage-4 contrastive sentence → add "contrasts-with" relationship.

4. **Hierarchical:** If A's definition contains B's name → add "builds-on" relationship (B → A).

5. **Temporal/causal:** If source text contains pattern `"[A] developed/evolved/emerged from [B]"` or `"[B] led to [A]"` → add "follows-from" relationship.

Generate labels using templates from a small label library:
```
"overlaps-with": "{A} and {B} both address {shared-keyword}"
"discussed-together": "{A} is explored alongside {B}"
"contrasts-with": "{A} stands in contrast to {B}"
"builds-on": "{A} builds on the foundation of {B}"
"follows-from": "{A} emerges from {B}"
```

**Output:** `Relationship[]`.

**Input:** Enriched concepts + relationships + normalized source text.

This is where most deterministic engines fail. You will not fail. Implement all 12 methods exactly.

**For each question type, the generation flow is:**
1. Select target concept(s)
2. Choose stem template from `question-stems.json`
3. Fill template slots
4. Generate 3 wrong answers using the method-specific distractor algorithm
5. Validate output via Section 5 gates

**Method 1: Definition Match (easy)**
- Stem: select from `easy-definition` stems, fill with concept name
- Correct: concept's `core` (may be paraphrased via synonym substitution from `synonyms.json` for variety — pick 1-2 content words, swap with synonyms, validate result still makes sense)
- Distractor 1: another concept's `core` (borrowed whole)
- Distractor 2: target concept's `core` with one key noun swapped for its antonym (from `antonyms.json`)
- Distractor 3: hybrid — first half of target's core + second half of a different concept's core

**Method 2: Philosopher Attribution (easy)**
- Find concept with associated philosopher (extracted via `philosophers.json` match in definition)
- Stem: `"Which philosopher is most associated with {concept}?"`
- Correct: the philosopher
- Distractors: 3 other philosophers from the same tradition set who appear in the source

**Method 3: Compare/Contrast (medium)**
- Requires a concept with ≥ 1 `opposites`
- Stem: `"How does {A} primarily differ from {B}?"`
- Correct: the actual distinction (from contrastive sentence detected in Stage 4)
- Distractor 1: reverse of the distinction (swap which concept has which property)
- Distractor 2: "They are essentially the same approach"
- Distractor 3: a difference pulled from a third unrelated concept pair

**Method 4: Application Scenario (medium)**
- Stem: `"A {role} who {action-derived-from-concept} is most likely applying which framework?"`
- `action-derived-from-concept` is constructed by extracting the main verb phrase from the concept's `core` and generalizing it to a scenario
- Correct: concept's category/framework
- Distractors: other categories/frameworks from the chapter

**Method 5: Negation Check (medium)**
- Stem: `"Which of the following is NOT characteristic of {concept}?"`
- Correct (the "NOT" option): a characteristic borrowed from a DIFFERENT concept
- Distractors: 3 actual characteristics of the target concept (extracted from `keywords` and core)

**Method 6: Fill the Framework (medium)**
- Requires a concept with a parent framework in hypernym data
- Stem: `"Within {framework}, {blank} plays the role of {role}."`
- Correct: the target concept
- Distractors: other concepts from different frameworks

**Method 7: Cause/Effect (medium)**
- Requires source text containing conditional structure near the concept
- Extract `if X then Y` or `because X, Y` patterns from source
- Stem: `"According to {concept}, what follows from {premise}?"`
- Correct: the stated consequence
- Distractors: consequences from other concepts + the inverse of the real consequence

**Method 8: Objection Match (hard)**
- Requires a concept with objections in source text
- Extract objection sentences: sentences near concept containing "objection", "criticism", "problem", "weakness", "however", "critics argue"
- Stem: `"Which of the following is a common objection to {concept}?"`
- Correct: the extracted objection
- Distractors: objections to other concepts + one of the target concept's own strengths (inverted into an objection-shaped sentence)

**Method 9: Example Identification (medium)**
- Stem: `"Which scenario best illustrates {concept}?"`
- Correct: a paraphrase of an example found in the concept's detail
- Distractors: examples from other concepts

**Method 10: Sequence/Process (hard)**
- Only usable when concept involves ordered steps (detected via numbered lists, "first...then...finally", ordinal words)
- Stem: `"In {concept}, what step follows {previous-step}?"`
- Correct: the actual next step
- Distractors: other steps out of order

**Method 11: Terminology Precision (hard)**
- Targets concepts with technical terms that have everyday meanings
- Stem: `"In the context of {concept}, what does '{term}' specifically mean?"`
- Correct: the technical definition from source
- Distractors: the everyday meaning + a plausible-sounding but wrong technical definition + a definition from a related concept

**Method 12: Synthesis (hard — Boss Fight only)**
- Requires at least one `contrasts-with` relationship
- Stem: `"How would {philosopher_A} respond to {philosopher_B}'s claim that {claim}?"`
- Correct: derived by combining philosopher_A's core principles with a rebuttal template
- Distractors: misattribute positions, oversimplify, reverse the philosophers

**Distractor validation (applies to ALL methods):**

Every set of 4 options must pass:
```typescript
function validateOptions(correct: string, distractors: string[]): boolean {
  const all = [correct, ...distractors];
  
  // All must be distinct (trigram Jaccard < 0.7)
  for (let i = 0; i < all.length; i++) {
    for (let j = i+1; j < all.length; j++) {
      if (trigramJaccard(all[i], all[j]) >= 0.7) return false;
    }
  }
  
  // All must be complete sentences or sentence fragments of similar length
  const lengths = all.map(a => a.length);
  const maxLen = Math.max(...lengths);
  const minLen = Math.min(...lengths);
  if (maxLen > minLen * 3) return false; // one option shouldn't be 3× longer than another (gives it away)
  
  // No option can be empty or too short
  if (lengths.some(l => l < 15)) return false;
  
  // No option can contain raw template slots
  if (all.some(a => /\{[^}]+\}/.test(a))) return false;
  
  // No option can be nonsensical (must have at least one content word besides stopwords)
  if (all.some(a => contentWordCount(a) < 2)) return false;
  
  return true;
}
```

**If distractor validation fails 3 times for a given question template, move to the next template. If all templates fail, skip that question. Do not include failed questions in the output bundle.**

## 5. QUALITY GATES (THE REJECTION LOOP)

Every stage has gates. Every artifact has gates. Every batch has gates. This is how you approximate LLM-level plausibility judgment.

### 5.1 Sentence-Level Validation

A generated sentence is valid only if:

```typescript
function isValidSentence(s: string): boolean {
  if (!s || s.length < 10) return false;
  if (s.length > 500) return false;
  
  // Must start with capital letter
  if (!/^[A-Z"]/.test(s.trim())) return false;
  
  // Must end with sentence punctuation
  if (!/[.!?"]$/.test(s.trim())) return false;
  
  // Must contain a verb (checked against common verb list)
  if (!containsVerb(s)) return false;
  
  // Must have stopword density ≤ 60% (otherwise it's nearly all function words)
  const tokens = s.toLowerCase().split(/\s+/);
  const stopwordCount = tokens.filter(t => STOPWORDS.has(t)).length;
  if (stopwordCount / tokens.length > 0.6) return false;
  
  // Must have lexical diversity ≥ 0.5 (unique/total ratio)
  const uniqueRatio = new Set(tokens).size / tokens.length;
  if (uniqueRatio < 0.5) return false;
  
  // Must not contain raw template slots
  if (/\{[^}]+\}/.test(s)) return false;
  
  return true;
}
```

### 5.2 Concept-Level Validation

A concept is valid only if:

```typescript
function isValidConcept(c: Concept): boolean {
  // Name checks
  if (c.name.length < 3 || c.name.length > 60) return false;
  if (CANVAS_CHROME.has(c.name.toLowerCase())) return false;
  if (STOPWORDS.has(c.name.toLowerCase())) return false;
  if (/^[\d\s\W]+$/.test(c.name)) return false;
  if (/^(the|a|an|this|that|these|those)\s/i.test(c.name)) return false;
  
  // Core checks
  if (!isValidSentence(c.core)) return false;
  
  // Core must actually discuss the concept (contain the concept name or a stem)
  const conceptStem = stem(c.name.toLowerCase());
  if (!c.core.toLowerCase().includes(conceptStem.substring(0, Math.min(5, conceptStem.length)))) {
    // Allow if the core is a definition that introduces the concept (starts with the concept name)
    if (!c.core.toLowerCase().startsWith(c.name.toLowerCase())) return false;
  }
  
  // Detail must not be near-duplicate of core
  if (c.detail && trigramJaccard(c.detail, c.core) > 0.8) return false;
  
  return true;
}
```

```typescript
async function generateWithRetry<T>(
  generator: () => T,
  validator: (t: T) => boolean,
  maxAttempts: number = 5
): Promise<T | null> {
  for (let i = 0; i < maxAttempts; i++) {
    const candidate = generator();
    if (validator(candidate)) return candidate;
  }
  return null; // explicit failure — do not return invalid content
}
```
